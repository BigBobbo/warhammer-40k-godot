## Summary findings

### Top 3 launch-blocker terrain gaps

1. **Aircraft / Towering wall-LoS exception is not honoured** (`40k/autoloads/EnhancedLineOfSight.gd:381-390`).
   The post-loop wall fall-back iterates every wall of every terrain piece and evaluates `_segment_intersects_wall` regardless of shooter / target keywords. Polygon-level Aircraft and Towering exceptions correctly bypass the polygon LoS branch but the wall fall-back ignores them, so:
   - An AIRCRAFT model CANNOT see through any ruin wall.
   - A TOWERING model cannot see over walls of other ruins (it only sees out of the ruin it stands in).
   Live-confirmed via `_check_single_line_of_sight((720,200),(900,200), …, {keywords:["AIRCRAFT"]}) → has_los:false, blocking_terrain:["ruins_1_wall"]`. Per Wahapedia 10e Aircraft: "visibility to and from AIRCRAFT models is determined normally" — the whole ruin (walls included) must be transparent for AIRCRAFT. Per Towering: "models with this keyword can draw line of sight over walls of Ruins."

2. **`state.board.terrain` is unpopulated, so the impassable-terrain placement check is a permanent no-op** (`40k/phases/MovementPhase.gd:6095-6120`, `40k/autoloads/GameState.gd:39`).
   `_position_intersects_terrain` reads `game_state_snapshot.board.terrain` (an array initialised to `[]` and never written by any code path) and only ever rejects pieces of `type: "impassable"`. Real terrain lives in `state.board.terrain_features` (or `TerrainManager.terrain_features`). Combined with the layouts' lack of any `type:"impassable"` pieces, this means **a model can be placed inside any ruin / wood polygon during deployment / movement / charge end-position validation without rejection**. The OA-28 Clankin' Forward and OA-29 Stompin' Forward "ignore terrain ≤4"" guard rails are also unreachable for the same reason. The fix is to read from `terrain_features` (with a `type` filter for genuinely-impassable pieces) AND ensure layouts include impassable terrain types where appropriate.

3. **Terrain category coverage is one-and-a-half categories deep.**
   Of the seven 10e Terrain Feature categories (Hills, Obstacles, Ruins, Woods, Craters, Sand Dunes, Barricades), only **two** appear in shipped data: `ruins` (8 layouts × 12 pieces each) and `woods` (only in `layout_2`, 2 pieces). Cover code, barricade engagement-range (1"→2"), and Difficult Ground penalties for crater / hill / obstacle types are written but never exercised in any shipped layout. Layouts 1, 3-8 contain only `type:"ruins"` pieces, even where the source GW Chapter Approved diagrams show woods, craters, and obstacles. This is a content-data gap, not an engine gap, but blocks any mission depending on non-ruin cover (e.g. matched-play scenarios that require area-terrain woods).

### Top 3 invisible features

1. **Multiple LoS algorithms diverge** — four implementations (`LineOfSightManager`, `EnhancedLineOfSight`, `LineOfSightCalculator`, `RulesEngine._check_legacy_line_of_sight`). They are NOT functionally equivalent: the legacy fall-back path (`RulesEngine.gd:3788-3886`, used by `_check_line_of_sight` when no model dicts are passed) does not consider walls AT ALL. The 'G' debug overlay (`LineOfSightManager.gd`) does check walls but inside the ruins branch only. `LineOfSightCalculator` is the AI scoring path. They will produce different visibility for the same pair of positions on the same board; whichever is hit first depends on the call site. Players cannot see this; it manifests only as engine-vs-overlay disagreements.

2. **`recommended_deployments` metadata present but invisible** — every `layout_*.json` declares which deployment maps it works best with (`recommended_deployments`), and `TerrainManager._preload_layout_metadata` (`:50-69`) loads it. But `MainMenu.gd:193-195` populates the dropdown from a hard-coded `terrain_options` list and never reads `_layout_metadata`. The recommendation is stored, queryable, and never shown. Listed in `TERRAIN_LAYOUTS_AUDIT.md` "Phase 4: Polish" — outstanding.

3. **Barricade engagement-range (1"→2") code path is dead in shipped data** — `TerrainManager.is_barricade_between` / `get_engagement_range_for_positions` / its 12 call sites in ChargePhase, FightPhase, RulesEngine all type-check on `type:"barricade"`. **No layout file places any piece with `type:"barricade"`.** The feature is wired engine-deep, totally invisible to players.
