> **Headline:** core charge mechanics & all 2026-05-verified items still work. The single largest **launchability** gap is **charge-roll modifiers (+N to charge / re-roll charge stratagems)**: the FactionStratagemLoader's `_map_effects` does not parse "add N to Charge rolls" or "re-roll Charge rolls" stratagem text, so 16+ faction stratagems (Tide of Muscle, Shock Assault, Reavers' Haste, From All Sides, Furious Dedication, Da Big Hunt, Stimulated Bio-Surge, Tireless Fervour, Spreading Madness, Hive Sight, Tide of Muscle, Subterranean Assault, Refusal to be Outdone, Euphoric Inspiration, Suppress and Overwhelm, Divine Imperative, Portal of Spite, **and the only charge-relevant Ork stratagem in the running session: 'ERE WE GO**) are loaded into the StratagemPanel but disabled with the rejection reason "X is not yet mechanically implemented." The charge dice pipeline (`_resolve_charge_roll`, `_is_charge_roll_sufficient`) reads `total_distance = rolls[0] + rolls[1]` directly with **no modifier hook** at any layer. This is the audit's required watch-item.

## Top 3 launch-blocker Charge gaps

1. **Charge-roll modifiers (+N, re-roll) not wired into the dice pipeline.** The single rule the audit prompt explicitly flagged ("Watch for: charge-roll modifiers (re-roll-charge stratagems, +1 charge auras) that may not be wired"). 13+ faction stratagems & at least 1 detachment ability boil down to `custom:unmapped` and are auto-rejected. For Orks/Custodes/Space Marines (the three rosterable factions), this affects 'ERE WE GO, Furious Dedication (+2), Shock Assault (re-roll), and Tide of Muscle (+1 + conditional re-roll). Fix scope: extend `FactionStratagemLoader._map_effects` with two regex/text matches → emit `EffectPrimitivesData.PLUS_CHARGE` (new constant) and `REROLL_CHARGE` (existing); add a `total_modifier` accumulator in `_resolve_charge_roll` that sums per-unit `effect_plus_charge` flags before computing `total_distance`. The reroll-side already has the consumer (`has_effect_reroll_charge` → `_check_command_reroll_or_resolve`) wired for abilities; faction stratagems just need the loader-side text-to-effect mapping.
2. **Battle-shock → cannot-charge gating relies on a separate flag set elsewhere.** `phases/ChargePhase.gd:1297-1372 _can_unit_charge` does not check `flags.battle_shocked` directly; it only honours `flags.cannot_charge`. If `BattleShock` autoload sets only `flags.battle_shocked` (without `cannot_charge`), shocked units could still charge. **Action:** spot-check `40k/autoloads/BattleShock.gd` (out of scope here; flag for cross-cutting audit `12_battle_shock_cascade.md`).
3. **Mid-path validation gaps (non-target ER + walls + model-through-model).** `_validate_engagement_range_constraints`, `_validate_no_model_overlaps`, and the wall-blocking check all run on **end positions only**. ChargeController prevents these client-side during drag, but a scripted/multiplayer-spoof client could submit `APPLY_CHARGE_MOVE` with a curved path that violates these rules. End-game effect: not a launchability crisis, but a multiplayer-fairness risk. Cite: `CHARGE_PHASE_AUDIT.md` §5 (still OPEN).

## Top 3 invisible features

1. **All faction +N/re-roll-charge stratagems show in the StratagemPanel but the Use button is greyed with reason "X is not yet mechanically implemented".** The visible affordance exists (StratagemPanelButton in Main.tscn:63, keyboard `S` to toggle, populate by phase) — but the player sees something they can never press. Classic invisible-feature inversion: the button is visible, the text says "ELIGIBLE-when-implemented", but the implementation is absent. This is more confusing than hiding the row entirely.
2. **`APPLY_CHARGE_MOVE` server-side wall-blocking.** ChargeController has `Measurement.model_overlaps_any_wall` in its drag-time validator (`scripts/ChargeController.gd:2581-2583`), so a player who drags through a wall sees a red ghost. But the server-authoritative `_validate_charge_movement_constraints` chain in `phases/ChargePhase.gd:1501-1583` does not call any wall-validator. End-state code-path "exists" (Measurement function is C-depth) but is "wired" only on the client (W-depth on client, **no W-depth on server**).
3. **`StratagemPanel.populate(player, phase_id)` accepts a `phase_id` parameter but ignores it.** Designed for phase-specific stratagem filtering (`scripts/StratagemPanel.gd:46`); only used to set the title. The signature suggests a feature that is C/W but not U-reachable. Player-facing impact is mild (other stratagems are merely greyed) but it's a code-shape mismatch.

## NEW launch-blocker findings (filed by this audit)

Need new effect-type constant in `40k/autoloads/EffectPrimitives.gd` (e.g. `PLUS_CHARGE`) plus `_apply_stratagem_effects` handler that sets a unit flag, plus `phases/ChargePhase.gd:_resolve_charge_roll` reader that adds the modifier to `total_distance` before the sufficiency check. Since the reroll-side already has full plumbing for abilities (`UnitAbilityManager.gd:174-182 Swift Onslaught`), only the loader-side mapping has to land for re-rolls; +N path needs both loader + new effect-flag.
