## Top 3 launch-blocker core-concept gaps

1. **No active-player-resolves-first ordering for simultaneous out-of-phase abilities** (`RulesEngine.gd`, `StratagemManager.gd:51-53` only has a re-entrancy guard). The Wahapedia rules-commentary "If two or more rules occur at the same time, the active player decides the order" is not implemented. Real-world impact: two units with Deadly Demise destroyed by the same attack resolve in iteration order, not active-player order. Two players holding stratagems triggering at the same moment have no priority resolution. **Acceptable for casual play; launch-blocker for tournament play.**

2. **DEVASTATING WOUNDS / HAZARDOUS mortal-wound spillover may violate the Wahapedia exception** (`apply_save_damage:9742-9748` calls `_apply_damage_to_unit_pool` for DW, which spills like ordinary mortal wounds). Per Wahapedia core rules, the exception "with HAZARDOUS or DEVASTATING WOUNDS abilities, each time those mortal wounds are allocated to a model, if that model is destroyed as a result of those mortal wounds, the remaining mortal wounds from that attack are lost" is **not** enforced — DW currently spills exactly like other mortal wounds. A 1×D6=6 DW vs five 1W boyz would kill all 5; per the FAQ exception only 1 should die and 5 damage be lost. **Needs Wahapedia citation verification + a spec test (`test_devastating_wounds_no_spillover.gd`) before fix.**

3. **Cover save 3+ cap keyword scope is unverified against Wahapedia** (`_calculate_save_needed:3704-3706`). The current code (universal cap, since 2026-05-05 commit `6958cff`) was a deliberate change from a previously keyword-gated implementation (INFANTRY/BEAST/SWARM). Memory entry `project_audit_2026_05` cites the keyword-gated version as verified. The most commonly cited Wahapedia text uses INFANTRY/BEAST/MOUNTED. **Wahapedia core-rules page truncates before this section on every fetch this session — human verification required.** No roster impact today (no SWARM models in `armies/*.json`); future faction support (Tyranid Ripper Swarms) will hit this.

## Top 3 invisible features

1. **`flags.save_modifier` (Mathhammer-only)** is wired into auto-resolve shoot, auto-resolve melee, and overwatch (clamp to ±1, applied to roll), but the **interactive** save flow ignores it — `WoundAllocationOverlay._roll_save_for_model:814-828` rolls a die and compares to `save_needed` without reading the flag. `prepare_save_resolution:9165` doesn't fold it into `save_needed` either. Today no production code sets `flags.save_modifier`, so the divergence is latent — but any future "-1 to enemy save" stratagem will silently no-op when the player uses interactive saves (the default for human-vs-human). **Either: (a) read the flag in `prepare_save_resolution` and fold it into `save_needed` once, or (b) read it in the overlay rolling step.** Option (a) is cleaner.

2. **Dead `_apply_damage_to_unit:9103`** (~50 lines, naive index-based allocation that does NOT prioritize wounded models) has zero callers. If a future maintainer accidentally wires it in as a "simpler" damage helper, regular damage allocation would silently violate the wounded-model-first rule. **Action: delete the function.** No caller search returned hits; safe to remove.

3. **Mixed-save defender choice not surfaced** (`_calculate_save_needed:3719-3722`). Code auto-picks the better save (lower of armour vs invuln). The Wahapedia rule allows the defender to choose either — relevant when, e.g., a defender wants to take the WORSE save to preserve a once-per-battle invuln stratagem trigger or to fail deliberately for a "when this model is destroyed" effect. **Not visible in any UI.** Add a defender choice toggle in `WoundAllocationOverlay` for the rare cases this matters; until then the engine plays optimally on behalf of the defender.

## Cross-phase divergences (Shooting vs Fight) — required focus per audit prompt

The audit prompt called out: "Highlight any case where the engine evaluates rules differently between Shooting and Fight phases (a single helper should resolve the attack sequence for both — flag if duplicated)." Findings:

1. **No shared resolver.** Hit→wound→save is implemented four times (interactive shoot, auto-resolve shoot, overwatch, melee) totalling ~3,300 lines. See "Hit/wound/save resolution code is shared" row above. **🐛 flagged.**

2. **Save modifier flag handling differs.** Auto-resolve shoot (`:3074`), auto-resolve melee (`:8597`), and overwatch (`:1183`) all read `flags.save_modifier` and clamp ±1. Interactive shoot/melee (via `prepare_save_resolution` → `WoundAllocationOverlay`) ignore it. **⚠️ flagged.**

3. **Critical-hit threshold check has a redundant guard in melee but not shoot.** Melee `:8294`: `unmodified_roll >= melee_crit_threshold or unmodified_roll == 6 or roll >= attack_ws`. Shoot `:2516`: `unmodified_roll >= critical_hit_threshold or final_roll >= attack_bs`. The `or unmodified_roll == 6` in melee is always already satisfied by the threshold check (threshold ≤ 6) — defensive but not buggy. Inconsistency only.

4. **Conversion X+ (lower critical-hit threshold for ranged at ≥12") has no melee analogue.** `get_critical_hit_threshold:5997` only branches for ranged distance — Wahapedia's Conversion is a ranged-only weapon ability, so this is correct, but the duplication of three resolvers makes it easy for someone adding a melee Conversion-equivalent to miss this branch.

5. **Tank Hunters (+1 hit/+1 wound vs Monster/Vehicle) is ranged-only;** Monster Hunters (re-roll hit vs Monster/Vehicle) is melee-only. Both correctly faction-gated by `has_tank_hunters_vs_target:5473` (ranged) and `has_monster_hunters_vs_target:5495` (melee). No bug.

6. **DA BIGGER DEY IZ (OA-49)** (conditional damage bonus) is melee-only and applied in `apply_save_damage:9654-9656` via `save_data.da_bigger_damage_bonus`. The shooting path's `prepare_save_resolution` doesn't surface that key — but the ability is melee-only so this is correct.

7. **Damage allocation rules (one-at-a-time, wounded-first) are correctly shared** via `_find_allocation_target_model:10080` and `_apply_damage_to_unit_pool:9870` / `_apply_damage_per_wound_no_spillover:9920`. Mortal wounds use the spillover variant; regular damage uses the no-spillover variant. **No divergence here.**

**Net assessment of Shooting vs Fight:** the rules themselves are evaluated equivalently (no rule asymmetry detected), but the **code structure** is duplicated to a degree that future divergences are likely. Refactor to a shared `_resolve_attack_sequence` helper is recommended (estimated 3-5 day effort) — would prevent future regressions like the `save_modifier` interactive-vs-auto-resolve gap from re-occurring as new abilities are added.
