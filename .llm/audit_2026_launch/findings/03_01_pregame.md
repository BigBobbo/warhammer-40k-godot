# 03.01 — Pre-game & Setup audit

**Source rules:** Wahapedia 10e Core Rules (https://wahapedia.ru/wh40k10ed/the-rules/core-rules/) + Chapter Approved 2025-26 (`40k/deployment_zones/Chapter Approved 2025-26.htm`, fetched via Wahapedia 2026-05-06). Key sections used: Strategic Reserves (`#Strategic-Reserves`), CA 25-26 deployment procedure (deploy alternation, terrain rule, attacker/defender roll-off, first-turn roll-off), Reinforcements step rules (cannot arrive turn 1).

**Codebase scope:** `40k/` only. Excluded `40k/.claude/worktrees/`.

**Live validation:** running Godot instance was mid-game (FIGHT phase, P2 active, Round 1) with another agent's session. ## Top 3 launch-blocker pre-game gaps

1. **No pre-deployment attacker/defender roll-off** (`autoloads/PhaseManager.gd:193-205`, `autoloads/TurnManager.gd:176-184`). Per CA 25-26 there are **two** roll-offs — step 1 (attacker/defender, before deployment) and step 12 (first turn, after Scouts). The codebase has only one (after deployment) and conflates the two outcomes. Consequence: defender-deploys-first cannot be implemented; `RedeploymentPhase.gd:11,32-33`'s "Attacker resolves first" assumption is incorrect (Player 1 is hard-coded as attacker).

2. **Deployment alternation always starts with Player 1** (`autoloads/TurnManager.gd:181-184`). 10e CA 25-26 mandates the **defender** deploys first; alternation must follow from that. Fix is gated by issue #1 — once attacker/defender is determined pre-deployment, set the initial active player to the defender.

3. **No battle-size selection / no battle-size enforcement** (`scripts/MainMenu.gd:38-68` has no bracket dropdown; `autoloads/ArmyListManager.gd:1208` lists points sizes but does not key them to brackets). The Strike-Force-only enhancement cap (≤3 enhancements at SF, 0 at Combat Patrol/Incursion, no cap at Onslaught) cannot be enforced. Players using the enhancement validator for matched-play can over-take enhancements without warning.

## Top 3 invisible features

1. **`DESIGNATE_WARLORD` action has no UI button** (`phases/FormationsPhase.gd:113,1108-1121` exposes it; `scripts/FormationsDeclarationDialog.gd` has no warlord section — verified by `grep` returning zero matches in `40k/scripts/`, `40k/dialogs/`, `40k/scenes/`). Multi-CHARACTER armies will hit "no Warlord designated" on `CONFIRM_FORMATIONS` with no way to fix it from the dialog. Auto-designate at `FormationsPhase.gd:756-766` handles single-CHARACTER rosters but not multi-character ones.

2. **Strategic Reserves unit-count cap** (50% units) is enforced in code (`DeploymentPhase.gd:396-402`, `FormationsPhase.gd:380-386`) but the FormationsDeclarationDialog reserves-section UI (`scripts/FormationsDeclarationDialog.gd:215-296`) only shows the **points** budget — not the unit-count budget. A player declaring reserves can hit "exceeds 50% reserves unit limit" without seeing how many units are allowed.

3. **`SEND_TO_STRATEGIC_RESERVES` action** in `RedeploymentPhase.gd:86-87` is wired, but redeployment only triggers when a unit has the relevant ability (Red Corsairs, Phantasm). Unless a roster carries one of those abilities, the entire RedeploymentPhase auto-skips (`RedeploymentPhase.gd:47-51`). Consequence: most players never see the redeployment phase exists — fine for stock rosters, but masks any bugs in the redeploy code path (which has full UI in `Main.gd` per the action handlers).

## Other notable findings

- `Datasheets_leader.csv` is the older mirror copy (`.llm/audit_2026_launch/01_inventory.md` flag) — leader-attachment validity in `FormationsPhase.gd` does NOT consult that CSV. It uses the per-unit `meta.leader_data.can_lead` keyword whitelist baked into `armies/*.json` rosters. So the leader-attachment validator is roster-driven, not data-driven — refresh-of-CSV won't help unless the roster export pipeline re-sources it.
- `MissionManager._get_configured_mission_id()` defaults to `take_and_hold` if `meta.game_config.mission` is missing, with a re-fallback at `initialize_mission()` line 89 if the chosen mission ID is unknown. Robust to bad config.
- The `40k/deployment_zones/Chapter Approved 2025-26.htm` snapshot is the ground truth for mission-pack rules in this repo. Worth re-fetching at next data refresh; the live Wahapedia URL `https://wahapedia.ru/wh40k10ed/the-rules/chapter-approved-2025-26/` was reachable as of 2026-05-06.
