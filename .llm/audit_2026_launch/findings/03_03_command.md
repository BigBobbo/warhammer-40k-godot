## Wahapedia rules text (as quoted)

> COMMAND PHASE — Your Command phase splits into two steps: 1. Command  2. Battle-shock
>
> **1. Command** — "At the start of your Command phase, before doing anything else, **both players gain 1CP**. Command points are a strategic resource that you can spend during the battle to use Stratagems."
> "Then, if you have any other rules that need to be resolved in the Command phase, you do so now before progressing to the Battle-shock step."
>
> **CP cap** — "Outside of the 1CP players gain at the start of the Command phase, **each player can only gain a total of 1CP per battle round, regardless of the source.**"
>
> **2. Battle-shock** — "If the result is greater than or equal to the best Leadership characteristic in that unit, the test is passed; otherwise, the test is failed and, until the start of your next Command phase, that unit is Battle-shocked." Effects: OC of all models becomes 0; cannot be affected by controlling player's Stratagems; must take Desperate Escape tests if Falling Back. (Wahapedia core-rules page; same URL as above.)

The page makes no exception for the very first Command phase of the game. The Q3-2024 Core Rules update is the current canonical text; nothing on the live Wahapedia page references a "skip CP on round 1" provision.

## Top 3 launch-blocker Command-phase gaps

1. **CP-grant rule diverges from current Wahapedia text (rule #1, finding #1).** Code skips CP for the first-turn player on round 1 and never grants CP to the opponent in your Command phase. Wahapedia 10.5 text says **"both players gain 1CP"** at the start of every Command phase with no first-turn exception. Net effect at round 1: each player ends up with one fewer CP than the rule prescribes (skipped first command phase = 1 lost; opponent never gains in your Command phase = effectively the same total over the round, but the timing of CP availability for opponent stratagems is wrong — e.g., a Counter-Offensive in your Fight phase expects CP that should have been generated in your Command phase). PR #336 should be reverted or re-justified against the current rules text.
2. **Battle-shock flag mutations bypass network sync (rule #5, finding).** `_clear_battle_shocked_flags` (CommandPhase.gd:216) and `_resolve_battle_shock_test` (CommandPhase.gd:809, 818) write `unit["flags"]["battle_shocked"]` directly instead of going through `PhaseManager.apply_state_changes`. In multiplayer, the host's view of "this unit is battle-shocked" never propagates to the client (or vice-versa), and the cleared-at-start-of-next-Command-phase reset likewise won't sync. Cover-save / OC-0 / stratagem-block consequences will diverge between peers.
3. **Acts of Faith / Dark Pacts not implemented (rules #20-21).** Two of the most-used Command-phase faction systems (Sororitas Acts of Faith / Miracle Dice; CSM Dark Pacts) have zero implementation. Adding either faction as a playable roster requires building these from scratch. Filed because the prompt explicitly asks for "Faction CP-generating abilities (Sisters Acts of Faith, Imperial Agents, etc.)" coverage.

## Top 3 invisible features

1. **Infiltrator Comms Array (rule #19).** Datasheet ability defined in `UnitAbilityManager.gd:1050-1060` as a Command-phase 5+-roll-for-1-CP, but no executor for `regain_cp_on_5plus`. A player fielding an Infiltrator Sergeant with this wargear sees the ability text on the unit card but it never fires. Pin test at `tests/test_audit_already_done_pin.gd:817-819` would mask this — it only asserts dict shape, not behaviour.
2. **109 Command-phase stratagems in `Stratagems.csv`, only 12 core + faction-loader output reach the runtime (rule #18).** Players viewing the StratagemPanel see stratagems for their own detachment (and the core 12), but ANY stratagem outside the active detachment, even if catalog-described as Command-phase, is never available. This is by-design for the "active detachment" rule, but the proportion of stratagems the loader marks `implemented: false` (custom effects) is currently unmeasured here and is the stage 04 entity audit's job.
3. **CP-cap awareness in UI is selective.** UI labels in `ScoringController.gd:191, 337-338, 342`, `MissionDiscardDialog.gd:118`, and AI tooltips show "+1 CP" or "+0 CP (cap reached)" on cap-aware actions, but **the Movement-phase Thievin' Scavengers notification** (the only other current cap-affected gain) is text-only and shows "+1 CP" or "(cap reached — no CP gained)" via a popup notification. There is no consolidated player-facing "round bonus CP used: 0/1" indicator next to the CP counter, so a player about to use Thievin' Scavengers cannot tell from the HUD that they've already burned the cap on a discard earlier the same round.
